# X Wallet AppsFlyer 事件回传范围讨论

日期：2026-06-24
来源：用户与 Hermes Agent 对话整理
主题：AppsFlyer 事件是否应同步推送给 Google Ads / Meta Ads，以及“所有媒体源”是否会影响平台学习。

## 背景问题

用户的问题集中在 AppsFlyer 事件回传配置：AF 的所有事件是否都需要同步推送给 Google 和 Meta，还是应该只把各自来源渠道的事件推送给各自渠道。

## 讨论结论

1. 不应把 AppsFlyer 的所有事件无差别推给广告平台。应先筛选关键漏斗事件，再决定每个事件对 Google Ads / Meta Ads 的回传范围。
2. 对已筛选的中浅层核心事件，冷启动和验证期通常建议使用“所有媒体渠道，包括自然流量”，而不是只推“仅该合作渠道”。
3. 对深层、高价值、高敏感的业务结果事件，例如审批通过、授信额度、放款成功、放款金额，不应默认全媒体源全量推送，应单独评估合规、商业敏感性、事件量和优化用途。
4. 推送其他渠道或自然量的同类优质事件给 Google / Meta，通常不会让平台学习更慢；相反，在冷启动或 iOS 隐私限制下，可能帮助平台积累足够事件信号。
5. 真正可能拖慢或带偏学习的不是“事件来自其他渠道”，而是事件太浅、质量混杂、定义不一致、样本和目标客群不可复制，或把敏感深层事件无差别全量推送。

## 关于“其他渠道事件是否会让平台找不到对应客群”

广告平台不是只根据“这个事件原本归因给哪个渠道”来学习客群。Google / Meta 更关注：

- 事件用户是否能被平台识别或匹配，例如 GAID、IDFA、平台登录态、设备图谱、MMP/CAPI 匹配字段。
- 用户完成了什么事件。
- 用户在平台生态中的行为特征。
- 当前 campaign 的地域、受众、版位、出价和优化目标。

因此，即使某个用户来自自然量、Google、Meta、聚合平台或其他渠道，只要事件回传给某个平台且该平台能匹配到用户，它仍然可以把该用户作为“完成过注册/申请提交”等事件的训练样本。

限制在于：

1. 如果平台无法匹配该用户，事件的用户级学习价值会下降，可能只能用于汇总统计或建模补充。
2. 如果其他渠道用户画像在目标平台上难以复制，例如 Google Search 的高意图搜索人群不一定能在 Meta 信息流中高效复现，学习价值会打折。
3. 如果混入低质量渠道、激励流量、羊毛党、内部测试、代理异常流量或定义不一致的事件，会污染优化目标。

所以风险不是“另一个渠道完全找不到客群”，而是“学到的样本不适合这个渠道复制”或“样本质量混杂导致优化方向变差”。

## iOS / AEM 解释

iOS / AEM 阶段指 Meta iOS App 广告在 Apple 隐私限制下处理事件回传、归因和优化的场景。AEM 是 Meta 的 Aggregated Event Measurement，用于在 iOS 14+、ATT、SKAN 等限制下衡量和优化 iOS 转化事件。

在 iOS 场景下，IDFA 可用性下降，Meta 可用事件信号更少，事件回传可能延迟或聚合。因此在 Meta iOS App 广告冷启动、事件验证和 AEM 配置阶段，不宜过早把 AppsFlyer 事件来源限制为“仅 Meta 来源”。中浅层核心事件更适合先设置为“所有媒体渠道，包括自然流量”，待事件稳定、Meta 自身事件量足够后再考虑收紧。

## X Wallet 建议配置

| 事件层级 | Google Ads | Meta Ads | 说明 |
|---|---|---|---|
| first_open / install | 必须导入/回传 | 必须回传 | 基础归因和安装转化可见性 |
| app_open / session_start | 做再营销时回传 | 做再营销时回传 | 再互动、受众和衡量 |
| 注册完成 | 所有媒体源，包括自然量 | 所有媒体源，包括自然量 | 高频、低敏、适合早期学习 |
| 申请开始 | 所有媒体源，包括自然量 | 所有媒体源，包括自然量 | 中层意向信号，帮助模型学习 |
| 申请提交 | 初期所有媒体源，稳定后评估是否收紧 | 初期所有媒体源，尤其 iOS/AEM 阶段；稳定后评估 | 核心质量事件，需监控后续质量 |
| 审批通过 / 授信成功 | 谨慎测试；可考虑仅本渠道或不带金额 | 谨慎测试；可考虑仅本渠道或不带金额 | 高质量但低频、敏感 |
| 放款成功 / 金额 | 不默认全媒体源全量推 | 不默认全媒体源全量推 | 终局价值事件，涉及敏感商业和合规边界 |

## 操作原则

1. 中浅层、高频、低敏、定义稳定的事件：优先全媒体源，包括自然量，帮助平台冷启动和事件识别。
2. 深层、低频、高价值、高敏感、渠道质量差异大的事件：逐步测试，必要时只推本渠道或只推事件不推金额。
3. 不要把“发送筛选后的关键事件给两个平台”误解成“所有 AF 事件全量同步给两个平台”。
4. 不要过早用“仅该合作渠道”切碎中浅层事件信号，除非已有明确合规或数据污染原因。
5. 评估是否收紧时，应按渠道比较注册后申请率、申请后审批率、放款率、拒绝率、欺诈率、重复申请率和 funded-loan CPA。
